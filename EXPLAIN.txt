1. WHAT IS BUSINESS360?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Business360 is a SaaS (Software as a Service) platform — like Odoo or Salesforce.

  "SaaS" means: multiple companies (called "organizations" or "tenants") can
   sign up and use the software at the same time, each with their own private
   data and settings.

Think of it like an apartment building:
  - The building = Business360
  - Each apartment = one company's account
  - Each apartment has its own lock (separate database)
  - The landlord (admin) can manage all apartments

Core idea:
  A company signs up → creates an organization → installs the modules they
  need (like CRM, Inventory, HR) → and starts using them.

Current modules available (in /modules/):
  - CRM         (starter+) — track leads, deals, customers; syncs with Contacts on lead conversion
  - Contacts    (starter+) — dedicated address book: company directory + all contacts with positions
  - Inventory   (growth+)  — products, warehouses, stock levels, purchase orders
  - Accounting  (pro+)     — invoices (with line items), expenses, revenue tracking
  - Projects    (enterprise+) — project management with tasks, assignees, priorities
  - Reports     (enterprise+) — cross-module KPI dashboard, monthly P&L, alerts
  - Calendar    (starter+) — team calendar with events, date-range filtering, hook integration

  Adding a new module = drop a folder in /modules/ with a manifest.json +
  backend/router.ts. The system auto-detects it at startup and it appears in
  the App Store. Each module is fully standalone.

  ARCHITECTURE NOTE — Dynamic & Fail-Safe:
  - Module frontend pages live DIRECTLY in apps/web/src/app/dashboard/<key>/
    as standalone Next.js pages. They do NOT import from /modules/.
  - The /modules/ folder only needs manifest.json + backend/router.ts for
    API auto-discovery. Deleting a module folder only removes it from the
    App Store catalog and API — it does NOT break the web app.
  - The sidebar (AppSidebar.tsx) renders modules dynamically based on the
    org's installed module list. Known modules use labeled icons; any unknown
    installed module key automatically appears with a generic icon at
    /dashboard/<key>. Removing a key from MODULE_NAV just stops the locked
    padlock icon from showing for that module.

  Each module has its own rules.txt that documents: its exact folder structure,
  manifest rules, API endpoints, RBAC, frontend layout, and code quality checklist.
  Read modules/<key>/rules.txt before modifying or adding a module.

"Monorepo" means: one Git repository that holds multiple projects.
It's like one big folder that contains both the frontend and backend,
so they can share code easily.

6. AUTHENTICATION — HOW LOGIN AND IDENTITY WORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

When a user logs in:
  1. They submit email + password to the /login page
  2. The backend checks the password, then creates a JWT (a signed ticket)
  3. The JWT is saved in a cookie called "access_token" in the browser
  4. Every future request automatically sends that cookie to prove identity
  5. Protected pages check the cookie — if missing or invalid, redirect to /login

Key files:
  apps/web/src/lib/auth.tsx
    → The AuthContext and useAuth() hook
    → Wraps the whole app so any page can ask "who is logged in?"
    → Holds: user info, org info, login(), logout(), refresh() functions

  apps/api/src/middleware/requireAuth.ts
    → A "gate" on the backend
    → Every protected API endpoint runs this first
    → Reads the access_token cookie → decodes the JWT → attaches user to request

IMPORTANT QUIRK:
  After login or after creating an org, the code MUST call refresh() from
  AuthContext BEFORE navigating. Otherwise the dashboard thinks no one is
  logged in and sends them back to /login.

7. MULTI-TENANT SYSTEM — HOW EACH COMPANY GETS ITS OWN DATA
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

"Multi-tenant" means many organizations share the same app but have separate data.

How it works in Business360:
  - The MAIN database holds: users, orgs, subscriptions, installed modules
  - Each ORGANIZATION gets its own separate PostgreSQL database
  - That org's CRM data (leads, deals, customers) lives only in their database
  - Company A can NEVER see Company B's data

Key file:
  apps/api/src/middleware/tenantResolver.ts
    → Reads the orgId from the logged-in user's JWT
    → Looks up which database belongs to that org
    → Connects to that specific database for the request
    → Caches the connection so it doesn't reconnect every time (performance)

8. MODULE SYSTEM — HOW ADD-ON FEATURES ARE MANAGED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The "module system" lets organizations turn features on/off, like installing apps
on a phone. Every module is fully self-contained inside its own folder.

MODULE STRUCTURE (each module is standalone — backend + frontend together):

  /modules/<name>/
  ├── manifest.json          ← name, key, version, plan, routes, permissions,
  │                             store info (category, tags, features, rating)
  ├── package.json           ← declares @business360/module-sdk dependency
  ├── backend/
  │   └── router.ts          ← Express API routes for this module
  └── frontend/
      ├── layout.tsx          ← tab navigation bar
      ├── page.tsx            ← overview/dashboard page
      └── <tab>/page.tsx      ← one file per sub-page (leads, invoices, …)

  The frontend/ files use @/ imports (resolved by Next.js to apps/web/src/).
  apps/web/src/app/dashboard/<name>/ files are THIN RE-EXPORTS only — one line:
    export { default } from "../../../../../../modules/<name>/frontend/page";

HOW DYNAMIC DETECTION WORKS (no server restart needed):

  ADDING a module:
    1. Drop the folder in /modules/<newkey>/ with manifest.json + backend/router.ts
    2. Open the App Store → GET /api/store/catalog calls bootstrapModules() which
       re-scans /modules/ from disk → new module appears immediately in the store
    3. Install it → first API call to /api/<newkey>/... loads the router on demand

  DELETING a module:
    1. Remove the folder from /modules/
    2. Open the App Store → next catalog fetch re-scans → module disappears

  KEY FILES:
    apps/api/src/bootstrap/registerModules.ts
      → Scans /modules/*/manifest.json and registers each in the engine registry
      → Called at startup AND inside every GET /api/store/catalog request

    apps/api/src/engine/route_registry.ts
      → A SINGLE dynamic wildcard:  app.use('/api/:moduleKey', ...)
      → Runs requireAuth → resolveTenant → requireModule → dynamic router dispatch
      → Each module router is loaded from disk on first request, then cached in memory
      → Previously separate static registrations for each module — now one handler for all

    apps/api/src/routes/store.ts (GET /api/store/catalog)
      → Calls bootstrapModules() on every request so added/removed modules are live

Key installation routes:
  apps/api/src/routes/modules.ts
    → GET  /api/modules         → list all available modules
    → POST /api/modules/install → install a module for the current org
    → DELETE /api/modules/:key  → remove a module

NOTE: The function is called getAllManifests() — NOT loadAllManifests().

9. ROLES AND PERMISSIONS — WHO CAN DO WHAT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Business360 has TWO completely separate permission systems:

A) PLATFORM LEVEL — SaaS Admin

  isAdmin = true on the User record grants access to /sadmin only.
  This is a boolean field on User, NOT a company role.
  Platform admins manage all orgs, billing, marketplace globally.
  They are NOT members of any tenant organization.
  Enforced by: apps/api/src/middleware/requireSaasAdmin.ts
               → Checks req.user.isAdmin === true → 403 if false
  Route: /sadmin — completely isolated layout, sidebar, UI

B) TENANT LEVEL — Company Roles

  ROLE               RANK   WHAT THEY CAN DO
  ─────────────────  ─────  ───────────────────────────────────────────────
  owner                3    Full control: add/remove members, all modules,
                            API keys, webhooks, teams.
  manager              2    Org-wide: invite members, create teams, access
                            all installed modules.
  sales_leader         1    Leads sales team — module access via team only.
  inventory_manager    1    Leads inventory team — module access via team.
  accountant           1    Leads accounting team — module access via team.
  engineer             1    Leads engineering team — module access via team.
  service_agent        1    Leads service team — module access via team.
  member               0    Standard: access only modules assigned via team.

Module access model (team-based):
  • Owners (rank 3) and managers (rank 2) always have access to every module.
  • All other roles access a module only if they are a member of a team
    whose moduleKey matches that module.
  • Enforced by: apps/api/src/middleware/requireModule.ts
  • Configured by: Settings → Teams (create a team, link it to a module,
    then add members to that team)
  • Settings → Module Access tab now shows a read-only view of which teams
    are linked to each module.

How tenant roles are enforced:
  apps/api/src/middleware/requireRole.ts
    → Called like: requireRole("owner") or requireRole("manager")
    → Uses ROLE_RANK (owner=3, manager=2, specialized=1, member=0) to compare.
    → If user's rank is too low → returns 403 Forbidden

Adding members — two methods available:
  1. Direct add (owner only): POST /api/org/members
       → Creates user account with email + hashed password, assigns role, adds to org.
       → If email already exists on platform, existing account is added to the org.
       → UI: Settings > Members > "Add Member" card (visible to owners only)
  2. Email invitation (manager+): POST /api/org/invites
       → Sends invite link via email; expires in 7 days.
       → Recipient must click link and accept to join.
       → UI: Settings > Members > "Invite via Email" card

CRITICAL RULES:
  - Platform auth (isAdmin) and tenant auth (role) NEVER mix
  - Tenant JWTs cannot access /api/sadmin routes
  - Platform admins do not appear in any tenant member list
  - The /sadmin layout is 100% isolated from the /dashboard layout

14. KNOWN QUIRKS — THINGS THAT ARE DIFFERENT FROM NORMAL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

These are gotchas — things that look wrong but are correct on purpose:

  1. SHADCN BUTTON LINKS
     shadcn/ui uses @base-ui (not Radix). The Button component does NOT support
     asChild. To make a button that is also a link, do this:
       <Link className={buttonVariants({ variant: "default" })} href="/page">
     NOT: <Button asChild><Link>...</Link></Button>

  2. useSearchParams() NEEDS SUSPENSE
     Any component that calls useSearchParams() must be wrapped in <Suspense>.
     The fix: extract that component into a child component, then wrap it.

  3. Select.onValueChange RETURNS string | null
     Always guard with: onValueChange={(v) => v && doSomething(v)}

  4. REFRESH BEFORE NAVIGATE
     After login or org creation, always call refresh() FIRST, then navigate.
     If you navigate first, the auth context still has stale/null user data
     and the dashboard will kick the user back to /login.

  5. ALL MODULE LAYOUTS ARE "use client"
     The CRM, Inventory, Accounting, HR, Projects layouts all use usePathname()
     to highlight the active tab. All must be client components ("use client").

  6. DASHBOARD LAYOUT IS "use client"
     Uses useAuth() hook, which requires it to be a client component.

  7. PRISMA DATETIME NULL vs UNDEFINED
     Prisma DateTime fields with @default(now()) (like issueDate on Invoice)
     are non-nullable in the database. When patching, pass undefined to skip
     the field — do NOT pass null, which will cause a TypeScript type error.

BACKEND API (apps/api/)
────────────────────────────────────────────────────────────────────────────────

  apps\api\src\index.ts
    What:    The entry point — starts the Express server.
             Applies all global middleware (helmet, rate limiting, cors, cookies),
             then registers all route files (/auth, /orgs, /modules, /admin, /store),
             then starts listening on a port.

  apps\api\src\lib\prisma.ts
    What:    Creates and exports the shared Prisma database client.
             This client connects to the MAIN database (users, orgs, subscriptions).
             All backend code imports from here to query the main database.

  apps\api\src\lib\tenantDb.ts
    What:    Manages connections to each organization's private database.
             Contains getTenantClient(orgId) which:
               - Looks up the org's database URL
               - Creates a Prisma client for that specific database
               - Caches the connection so the same org doesn't reconnect every request

  ── MIDDLEWARE ──────────────────────────────────────────────────────────────

  apps\api\src\middleware\requireAuth.ts
    What:    A "gate" function placed before any protected route.
             Reads the "access_token" cookie from the request.
             Decodes and verifies the JWT inside.
             Attaches the user's id, email, role, and orgId to the request object.
             If the token is missing or invalid → returns 401 Unauthorized.

  apps\api\src\middleware\requireRole.ts
    What:    A TENANT role-check gate used AFTER requireAuth.
             Called as: requireRole("owner") or requireRole("manager")
             Uses ROLE_RANK (owner=2, manager=1, member=0) to compare.
             If user's rank is too low → returns 403 Forbidden.
             Does NOT handle platform (SaaS) admin — see requireSaasAdmin.ts.

  apps\api\src\middleware\requireSaasAdmin.ts
    What:    Platform admin guard — completely separate from requireRole.
             Checks req.user.isAdmin === true.
             If false → returns 403 immediately.
             Used only on /api/sadmin/* routes.

  apps\api\src\middleware\tenantResolver.ts
    What:    Connects the request to the correct org's private database.
             Reads orgId from the authenticated user (set by requireAuth).
             Calls getTenantClient(orgId) and attaches the tenant DB client
             to the request so route handlers can query org-specific data.

  apps\api\src\middleware\requireModule.ts
    What:    Checks that the org has a specific module installed before
             allowing access to that module's routes.
             Example: blocks CRM routes if the CRM module isn't installed.

  apps\api\src\middleware\errorHandler.ts
    What:    A global error catcher placed at the very end of the middleware chain.
             When any route throws an error, this catches it and returns a clean
             JSON error response instead of crashing the server.

  apps\api\src\middleware\requireApiKey.ts
    What:    Programmatic API access gate (alternative to cookie JWT auth).
             Reads Authorization: Bearer b360_... header. SHA-256 hashes the key,
             looks it up in the api_keys table (isActive=true), updates lastUsedAt
             non-blocking, then sets req.user with owner-level access for that org.

  apps\api\src\middleware\logActivity.ts
    What:    Fire-and-forget audit log helper.
             logActivity(req, action, entity, entityId?, meta?) writes a record
             to the tenant's activityLog table. Called inside route handlers after
             mutations (e.g., lead.create). Errors are swallowed — never blocks the response.

    What:    Notification CRUD (uses `prisma as any` with migration-pending guard).

  apps\api\src\services\webhookService.ts

MODULE PACKAGES (modules/)   ← plug-in folder: drop a folder here to add a module
────────────────────────────────────────────────────────────────────────────────

Each module folder is fully self-contained — backend AND frontend live together:

  modules\<name>\manifest.json
    What:    The module's single source of truth. Contains BOTH engine metadata
             (key, name, version, requiredPlan, routes, permissions) AND App Store
             display data (category, longDescription, tags, features, rating, price).
             The API reads this at startup; the store serves it to the frontend.

  modules\<name>\package.json
    What:    npm workspace descriptor. Declares @business360/module-sdk as a dep.

  modules\<name>\backend\router.ts
    What:    Express Router with all API endpoints for this module.
             Imports utilities from @business360/module-sdk.
             Mounted at /api/<key> automatically when the manifest is discovered.

  modules\<name>\frontend\page.tsx        ← module overview / dashboard page
  modules\<name>\frontend\layout.tsx      ← tab navigation layout (where applicable)
  modules\<name>\frontend\<tab>\page.tsx  ← one file per sub-page (leads, invoices, …)
    What:    The actual React component code for all this module's UI pages.
             Uses @/ imports (resolved by Next.js to apps/web/src/) so shadcn/ui
             components, API helpers, and utilities all work normally.
             apps/web/src/app/dashboard/<name>/ files are thin re-exports that
             simply forward to these module files — routing stays in Next.js,
             logic lives here.

  Pattern for apps/web re-export stubs:
    apps/web/src/app/dashboard/crm/page.tsx →
      export { default } from "../../../../../../modules/crm/frontend/page";

  To add a new module: drop a folder in /modules/ with manifest.json +
  backend/router.ts + frontend/ pages, then add matching re-export stubs in
  apps/web/src/app/dashboard/<name>/.

    Changes are saved to erpai/data/ai-config.json and take effect immediately
    (ERPAI reads the config file on every request)
    API: GET/PATCH /api/sadmin/erpai/config

  WHAT IT IS:
    A standalone Next.js app that acts as a 24/7 AI developer agent for
    Business360. It has deep knowledge of the codebase and can chat,
    analyze code by reading actual files, and manage development tasks.

  KEY FEATURES:
    1. AI Chat (/chat)
       - Streaming chat powered by Claude Opus 4.7 with adaptive thinking
       - Prompt caching: EXPLAIN.txt + project context cached each session
       - Persistent conversation history in the browser session
       - Suggests common questions about the codebase

    2. Code Analysis (/analyze)
       - Agentic loop: Claude reads real files, greps real code (no guessing)
       - Tools: read_file, list_directory, grep_code
       - Up to 12 tool-call iterations per analysis
       - Preset buttons: Security Audit, Find Missing Tests, Performance Review,
         Module Architecture, Dead Code, TypeScript Strictness

    3. Task Manager (/tasks)
       - Create, update, and delete development tasks
       - AI can suggest tasks based on a selected focus area
       - Persisted to erpai/data/tasks.json (no database needed)
       - Filter by status (todo / in_progress / done)
       - Task categories: bug, feature, refactor, test, docs

  CRITICAL-001: ERPAI Authentication
    All ERPAI routes (chat, analyze, tasks) were previously unprotected.
    Added apps/erpai/src/lib/requireAuth.ts using `jose` (jwtVerify) — the
    only JWT library that works in Next.js Edge + Node runtimes.
    Every API route handler now calls requireAuth(req) as the first step;
    returns NextResponse (401) if invalid, or { user: AuthPayload } on success.
    erpai/.env.local gets JWT_SECRET so it matches the main API token.

  CRITICAL-010: JWT Invalidation on Role Change
    New field `tokenIssuedBefore DateTime?` added to OrgMember in
    packages/db/prisma/schema.prisma.
    Migration: packages/db/prisma/migrations/20260510600000_token_issued_before/
    When a member's role is changed (PATCH /api/org/members/:id or /:id/role),
    tokenIssuedBefore is set to `new Date()`.
    requireAuth middleware (apps/api/src/middleware/requireAuth.ts) now does an
    async DB check: if the JWT's `iat` is before tokenIssuedBefore → 401
    "session invalidated". Fail-open on DB error (request passes through).

  DB-003: Float → Decimal for Financial Fields
    All financial Float fields in apps/api/prisma/tenant.prisma migrated to
    Decimal @db.Decimal(20, 4) to avoid floating-point rounding errors.
    Affected models: Deal.value, Product.unitPrice/costPrice,
    PurchaseOrder.totalAmount, PurchaseOrderItem.*,
    Invoice.subtotal/tax/total, InvoiceItem.*, Expense.amount, Employee.salary.
    Migration SQL: apps/api/prisma/migrations/tenant_financial_decimal.sql
    Prisma Decimal objects serialize as strings by default — fixed with a
    res.json() override middleware in index.ts that converts any object with
    .toNumber() to a plain JS number before sending to the client.
    packages/module-sdk/src/index.ts exports toNum() helper for safe arithmetic
    on Decimal values (used in accounting router stats queries).

  PRODUCTION HARDENING

  Pino Logging:
    apps/api/src/lib/logger.ts — pino instance with LOG_LEVEL env var.
      - Development: pino-pretty with colorized, human-readable output.
      - Production: structured JSON (Datadog / CloudWatch compatible).
    pino-http middleware in index.ts logs every request with method/url/status/
    response-time; /health and /api/health are excluded to reduce noise.

  Health Endpoints:
    GET /health and GET /api/health both return:
      { status, service, version, uptime (seconds), timestamp }
    The /health path is used by Docker healthchecks.

  Graceful Shutdown:
    SIGTERM and SIGINT handlers call shutdown(). Guards against double-shutdown
    with isShuttingDown flag. server.close() drains open connections; callback
    disconnects all Prisma clients (main + per-tenant). Force-exits after 10s.

  Docker / docker-compose:
    apps/api/Dockerfile  — multi-stage: base → deps (npm ci) → builder (tsc)
                           → runner (production, copies dist + generated + modules)
    apps/web/Dockerfile  — multi-stage for Next.js standalone output
    docker-compose.yml   — postgres (healthcheck), api (depends on postgres healthy),
                           web (depends on api healthy). All env vars via ${VAR:-default}.
    .env.example at repo root — documents all required + optional env vars.

  GitHub Actions CI (.github/workflows/ci.yml):
