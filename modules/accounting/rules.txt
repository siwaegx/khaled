MANIFEST RULES
--------------
- key must be exactly: "accounting"
- version: start "1.0.0"
- routes array must match every page in frontend/pages/
- permissions must cover all exposed operations
- roles in each permission: one of ["owner", "admin", "member"]
- accounting:reports:read — restrict to owner and admin only (financial sensitivity)


BACKEND RULES
-------------
- All routes protected by requireAuth — never skip
- Use tenant DB — never write financial data to the shared DB
- Logic in services/, not route handlers
- Return consistent JSON:  { data, error, message }
- HTTP status codes:
    200, 201, 400, 401, 403, 404, 500
- TypeScript strict — no "any"
- Max 300 lines per file

Entities and their API endpoints:

  Invoices
    GET    /api/accounting/invoices          → list invoices (filter by status, date range)
    POST   /api/accounting/invoices          → create draft invoice
    GET    /api/accounting/invoices/:id      → get invoice with line items
    PATCH  /api/accounting/invoices/:id      → update draft invoice
    POST   /api/accounting/invoices/:id/send → mark as sent (status: draft → sent)
    POST   /api/accounting/invoices/:id/pay  → mark as paid (status: sent → paid)
    DELETE /api/accounting/invoices/:id      → delete draft only (cannot delete sent/paid)
    Fields: id, number (auto-incremented), customerName, customerEmail, issueDate,
            dueDate, status, subtotal, tax, total, notes
    Line items: invoiceId, description, quantity, unitPrice, lineTotal

  Expenses
    GET    /api/accounting/expenses          → list expenses (paginated)
    POST   /api/accounting/expenses          → create expense
    PATCH  /api/accounting/expenses/:id      → update expense
    DELETE /api/accounting/expenses/:id      → delete expense
    Fields: id, description, amount, category, date, attachmentUrl?, createdAt

  Reports (read-only — aggregated queries)
    GET    /api/accounting/reports/summary   → revenue, expenses, profit (by month)
    GET    /api/accounting/reports/invoices  → invoice status breakdown
    GET    /api/accounting/reports/expenses  → expenses by category

  Business rules:
    - Invoice number must be auto-generated and sequential per org (acc-0001, acc-0002…)
    - Sent/paid invoices are immutable — reject PATCH
    - total = subtotal + tax (always recalculate server-side, never trust client total)

DATABASE MIGRATION (migrations/001_accounting_initial.sql)
----------------------------------------------------------
Runs inside TENANT database only.

Tables:
  acc_invoices      (id, number SERIAL, customer_name, customer_email, issue_date, due_date,
                     status, subtotal, tax, total, notes, created_at, updated_at)
                    status CHECK IN ('draft','sent','paid','overdue')
  acc_invoice_items (id, invoice_id REFERENCES acc_invoices, description, quantity,
                     unit_price, line_total)
  acc_expenses      (id, description, amount, category, date, attachment_url, created_at, updated_at)

Rules:
- id: TEXT PRIMARY KEY DEFAULT gen_random_uuid()
- Timestamps: TIMESTAMPTZ DEFAULT NOW()
- acc_invoice_items.invoice_id CASCADE DELETE
- No org_id column (tenant DB is org-scoped)
- Migration must be idempotent: CREATE TABLE IF NOT EXISTS


INSTALLATION FLOW
-----------------
1. Engine validates requiredPlan = "pro" against org.plan
2. Engine checks InstalledModule — rejects if already active
3. Engine runs 001_accounting_initial.sql on tenant DB
4. Engine saves InstalledModule { orgId, moduleKey: "accounting", isActive: true }
5. Routes registered under /api/accounting/*
6. Sidebar updated from manifest.routes

Uninstall:
- isActive = false (data preserved)
- Routes deregistered immediately
